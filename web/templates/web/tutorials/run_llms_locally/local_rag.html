Local RAG Pipeline

End-to-end RAG with local embeddings — chunking strategies, hybrid search, reranking, and running everything in Docker with zero cloud dependencies.

Series progress9 / 9

What makes a fully local RAG pipeline different

The LangChain RAG tutorial covered the standard pattern: OpenAI embeddings + FAISS + GPT-4. A fully local pipeline replaces every cloud component — embeddings, vector store, and the LLM — with local equivalents. No API keys, no internet, no per-token cost.

Architecture comparison

Cloud RAG

  • → text-embedding-3-small (OpenAI)
  • → Pinecone / Weaviate (cloud vector DB)
  • → gpt-4o-mini (OpenAI LLM)

Local RAG (this page)

  • → nomic-embed-text (Ollama)
  • → Chroma or FAISS (local)
  • → llama3.1:8b (Ollama LLM)

Step 1 — Ingestion (ingest.py)

The ingestion script loads documents, splits them into chunks, embeds each chunk, and persists the vector store to disk. Run this once whenever your source documents change.

bash 
pip install langchain-ollama langchain-community chromadb pypdf
python 
from pathlib import Path
from langchain_community.document_loaders import PyPDFLoader, DirectoryLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_ollama import OllamaEmbeddings
from langchain_community.vectorstores import Chroma

DOCS_DIR = "./docs"
CHROMA_DIR = "./chroma_db"
EMBED_MODEL = "nomic-embed-text"

def ingest():
    # Load all PDFs from docs/ directory
    loader = DirectoryLoader(DOCS_DIR, glob="**/*.pdf", loader_cls=PyPDFLoader)
    raw_docs = loader.load()
    print(f"Loaded {len(raw_docs)} pages from {DOCS_DIR}")

    # Split into overlapping chunks
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=512,
        chunk_overlap=64,
        separators=["\n\n", "\n", ".", " ", ""],
    )
    chunks = splitter.split_documents(raw_docs)
    print(f"Split into {len(chunks)} chunks")

    # Embed with local Ollama model
    embeddings = OllamaEmbeddings(model=EMBED_MODEL)

    # Persist to disk — reusable across runs
    vectorstore = Chroma.from_documents(
        documents=chunks,
        embedding=embeddings,
        persist_directory=CHROMA_DIR,
    )
    print(f"Stored {len(chunks)} chunks in {CHROMA_DIR}")

if __name__ == "__main__":
    ingest()
Pull the embedding model first: ollama pull nomic-embed-text — only needed once.

Chunking strategy matters

The chunk size you choose directly affects retrieval quality. There is no universally correct value — it depends on your document type and query style.

Chunk sizeBest forRisk
128–256 tokensPrecise fact extraction, Q&A over structured contentLoses context — answers lack surrounding explanation
512 tokensGeneral-purpose default — articles, documentation, PDFsMay split mid-sentence without good separators
1024+ tokensLong-form analysis, book chapters, legal documentsRetrieves irrelevant context alongside relevant content
Overlap is not optional. A chunk_overlap of 10–15% of chunk_size (e.g. 64 tokens for 512-token chunks) prevents key sentences from being split across chunks and never retrieved together.

Step 2 — Query pipeline (rag.py)

The query pipeline loads the persisted Chroma store, retrieves relevant chunks for the user's question, and passes them to the local LLM.

python 
from langchain_ollama import OllamaEmbeddings, ChatOllama
from langchain_community.vectorstores import Chroma
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

CHROMA_DIR = "./chroma_db"
EMBED_MODEL = "nomic-embed-text"
CHAT_MODEL = "llama3.1:8b"

# Load persisted store
embeddings = OllamaEmbeddings(model=EMBED_MODEL)
vectorstore = Chroma(persist_directory=CHROMA_DIR, embedding_function=embeddings)
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

# RAG prompt — instruct the model to use only the context
prompt = ChatPromptTemplate.from_messages([
    ("system", """You are a helpful assistant. Answer the question using ONLY the
context provided below. If the answer is not in the context, say so clearly.

Context:
{context}"""),
    ("human", "{question}"),
])

llm = ChatOllama(model=CHAT_MODEL, temperature=0)

def format_docs(docs):
    return "\n\n---\n\n".join(doc.page_content for doc in docs)

rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

if __name__ == "__main__":
    while True:
        q = input("\nQuestion (or 'quit'): ").strip()
        if q.lower() == "quit":
            break
        answer = rag_chain.invoke(q)
        print(f"\nAnswer: {answer}")

Hybrid search — combining BM25 and vector retrieval

Pure vector search misses exact keyword matches. Hybrid search runs BM25 (keyword) and vector retrieval in parallel, then merges results with Reciprocal Rank Fusion (RRF). This usually outperforms either method alone.

bash 
pip install rank-bm25 langchain-community
python 
from langchain_community.retrievers import BM25Retriever
from langchain.retrievers import EnsembleRetriever

# BM25 needs the raw text chunks (not the vector store)
bm25_retriever = BM25Retriever.from_documents(chunks)
bm25_retriever.k = 4

# Vector retriever from Chroma
vector_retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

# Ensemble: 40% BM25, 60% vector — tweak weights for your use case
hybrid_retriever = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.4, 0.6],
)

# Drop-in replacement for the single retriever in rag_chain
rag_chain = (
    {"context": hybrid_retriever | format_docs, "question": RunnablePassthrough()}
    | prompt | llm | StrOutputParser()
)

BM25 is particularly effective for product names, error codes, and proper nouns that don't embed well. Adding it costs minimal overhead because it operates on in-memory term frequencies.

Reranking — improving precision after retrieval

Retrieval returns the top-k candidates. A reranker re-scores them using a cross-encoder — more accurate than embedding similarity, but too slow to run on the full corpus. The pattern is retrieve-many, rerank-to-few.

bash 
pip install sentence-transformers
python 
from sentence_transformers import CrossEncoder
from langchain_core.documents import Document

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def rerank(query: str, docs: list[Document], top_n: int = 3) -> list[Document]:
    pairs = [(query, doc.page_content) for doc in docs]
    scores = reranker.predict(pairs)
    ranked = sorted(zip(scores, docs), key=lambda x: x[0], reverse=True)
    return [doc for _, doc in ranked[:top_n]]

# Retrieve 8, rerank to top 3 — keeps context window small
broad_retriever = vectorstore.as_retriever(search_kwargs={"k": 8})

def retrieve_and_rerank(query):
    docs = broad_retriever.invoke(query)
    return rerank(query, docs, top_n=3)

rag_chain = (
    {"context": (lambda q: format_docs(retrieve_and_rerank(q))),
     "question": RunnablePassthrough()}
    | prompt | llm | StrOutputParser()
)
cross-encoder/ms-marco-MiniLM-L-6-v2 — 23 MB, runs fast on CPU, no GPU needed. Larger cross-encoders exist but this is a good default for local pipelines.

Returning source citations

Always show which chunks were used to generate the answer. This lets users verify the response and builds trust — especially important for local models with higher hallucination rates.

python 
from langchain_core.runnables import RunnableParallel

rag_with_sources = RunnableParallel(
    {"answer": rag_chain, "source_docs": retriever}
)

result = rag_with_sources.invoke("What is gradient descent?")

print("Answer:", result["answer"])
print("\nSources:")
for doc in result["source_docs"]:
    src = doc.metadata.get("source", "unknown")
    page = doc.metadata.get("page", "?")
    print(f"  • {src} (page {page})")

Docker deployment

Running Ollama in Docker makes the pipeline reproducible and easy to share. Use the official Ollama image and mount your models directory so you don't re-download models on every container restart.

yaml 
services:
  ollama:
    image: ollama/ollama:latest
    ports:
      - "11434:11434"
    volumes:
      - ~/.ollama:/root/.ollama   # persist downloaded models
    restart: unless-stopped

  rag-app:
    build: .
    depends_on:
      - ollama
    environment:
      - OLLAMA_BASE_URL=http://ollama:11434
    volumes:
      - ./docs:/app/docs
      - ./chroma_db:/app/chroma_db
dockerfile 
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["python", "rag.py"]
bash 
# Start everything, pull models, then ingest
docker compose up -d
docker exec rag-app-ollama-1 ollama pull nomic-embed-text
docker exec rag-app-ollama-1 ollama pull llama3.1:8b
docker compose run rag-app python ingest.py
GPU in Docker: Add deploy.resources.reservations.devices with driver: nvidia to the ollama service to enable GPU passthrough. Requires the NVIDIA Container Toolkit.

Common failures and fixes

Model hallucinating despite context

Your system prompt must explicitly say "answer using ONLY the provided context". Many local models ignore soft instructions — be direct. Add "If the answer is not in the context, say you don't know."

Retrieval returns irrelevant chunks

Usually a chunking problem. Try reducing chunk_size from 1024 to 512. Add reranking. Check that your embedding model is appropriate for the language and domain of your documents.

Chroma persistence not loading

The persist_directory must be the same in both ingest.py and rag.py. Chroma v0.4+ handles persistence automatically — don't call .persist() manually (deprecated).

Slow ingestion with large document sets

Embedding is the bottleneck. Run ingestion in batches. Consider switching to all-minilm (23 MB, faster) for initial iteration, then switch to nomic-embed-text for production quality.

Series complete — quick summary

  • Full local RAG: OllamaEmbeddings + Chroma (persistent) + ChatOllama — zero cloud cost
  • Chunk size 512 with 10–15% overlap is a reliable default for mixed document types
  • Hybrid search (BM25 + vector + RRF) consistently outperforms vector-only retrieval
  • Reranking with ms-marco-MiniLM-L-6-v2: retrieve 8, pass top 3 to the LLM
