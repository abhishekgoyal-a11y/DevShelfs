Memory & Context

Short-term, long-term, and entity memory — what they do, when to enable them, and the caveats that bite people in production.

Series progress9 / 15
{# ── Content ── #}

Why Memory?

By default, every kickoff starts fresh — agents forget everything between runs. Memory persists facts across runs, so the same crew gets smarter over time without you re-feeding context.

The Three Kinds of Memory

Short-term (this run)

Recent task outputs and reasoning, available within a single kickoff. RAM-based, dies when the run ends.

Long-term (across runs)

Persisted to disk (SQLite by default). Stored as embeddings — agents can semantically recall facts from previous runs.

Entity memory

Tracks specific entities (people, products, projects) and their attributes across runs. Useful for CRMs, support bots.

Enabling Memory

Turn it on at the crew level — CrewAI handles all three kinds for you.

Memory-enabled crew

PYTHON 
from crewai import Crew, Process

crew = Crew(
    agents=[researcher, writer],
    tasks=[research_task, write_task],
    process=Process.sequential,
    memory=True,                       # enable short + long + entity
    verbose=True,
)

# First run: learns about Acme Corp
crew.kickoff(inputs={"company": "Acme Corp"})

# Second run, days later: still remembers Acme facts
crew.kickoff(inputs={"company": "Acme Corp", "question": "What did we learn last time?"})

Custom Embedder & Storage

The default uses OpenAI embeddings + a local SQLite store. For production, point at your own infra.

Custom embedder

PYTHON 
crew = Crew(
    agents=[...],
    tasks=[...],
    memory=True,
    embedder={
        "provider": "openai",
        "config": {"model": "text-embedding-3-small"},
    },
)

Passing Context Manually

Sometimes you don't want auto-memory — you want to inject specific context per run. Use the inputs dict.

Per-run context injection

PYTHON 
customer_history = load_customer_history(customer_id)

result = crew.kickoff(inputs={
    "customer_id": customer_id,
    "history": customer_history,    # appears anywhere {history} is referenced
    "current_ticket": ticket_body,
})

Caveats

  • Memory costs tokens. Recalled facts are stuffed into the prompt — watch your context usage.
  • Memory can mislead. Stale facts get recalled too. Add timestamps if that matters.
  • Test with memory off first. If your crew works deterministically, memory is just a bonus, not a fix.
