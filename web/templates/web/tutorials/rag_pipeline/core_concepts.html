Core Concepts

1. Embeddings: Text as Vectors

An embedding is a numerical representation of text. Instead of storing "The cat sat on the mat" as a string, we convert it to a list of numbers that capture its meaning.

Python 
Text: "The cat sat on the mat"
↓
Embedding: [0.23, -0.41, 0.15, 0.89, ..., 0.05]
           (1536 numbers for OpenAI's text-embedding-3-small)

Why embeddings? Because embeddings capture semantic meaning. Similar texts have similar embeddings, even if they use different words.

Example: "The cat sat on the mat" and "A feline rested on the carpet" have different words but similar embeddings because they mean the same thing. This is why embeddings enable semantic search.

Key Properties:

1

Fixed dimension: Always the same length (e.g., 1536). You can compute distances.

2

Semantic: Similar meanings → similar vectors. "Dog" is closer to "cat" than to "pizza".

3

Language-agnostic: Multilingual embeddings work across languages.

4

Deterministic: Same text always produces the same embedding (within the same model).

2. Vector Similarity: Finding Neighbors

Once you have vectors, you can measure how similar two pieces of text are by computing the distance between their embeddings.

Cosine Similarity (Most Common)

Measures the angle between two vectors. Range: -1 to 1 (but typically 0 to 1 for text).

Python 
similarity = (A · B) / (||A|| × ||B||)

Example:
Text A: "Python programming language"
Text B: "Python is great for coding"
Cosine Similarity: 0.92 (very similar!)

Text C: "What's the weather?"
Cosine Similarity to A: 0.15 (not similar)

Euclidean Distance (L2)

Straight-line distance between vectors. Lower is more similar. Slower but works well for some use cases.

Manhattan Distance (L1)

Sum of absolute differences. Used for sparse vectors. Faster but less accurate for dense embeddings.

For RAG: Use cosine similarity. It's the standard, fast, and works best with normalized embeddings. Most vector databases default to it.

3. Vector Databases: Searching at Scale

Brute-force similarity search is slow. With a million documents, comparing your query to every embedding takes seconds. Vector databases use smart indexing to search in milliseconds.

How Vector DBs Work:

1. Indexing: When you add documents, embeddings are indexed using algorithms like:

  • HNSW (Hierarchical Navigable Small World) — Fast, approximate, most popular
  • IVF (Inverted File) — Clusters vectors, faster for large datasets
  • LSH (Locality Sensitive Hashing) — For approximate nearest neighbors

2. Querying: When you search, the DB uses the index to find similar vectors without comparing all of them.

query_embedding = embed("How do I use Python?") # Instead of comparing to 1M docs, # find ~100 candidates using index # Then return top-10 by similarity top_10_docs = vector_db.search(query_embedding, k=10)

Speed tradeoff: Approximate search (HNSW) is much faster but might miss the truly best match. Exact search is slow. RAG typically uses approximate.

4. The RAG Architecture

Now let's see how all these pieces fit together:

Phase 1: Indexing (Offline, happens once or periodically)

Python 
Step 1: Load Documents
documents = ["tax_guide.pdf", "FAQ.html", "policies.txt", ...]

Step 2: Chunk Them
chunks = [
  "A tax deduction is...",
  "Common deductions include...",
  "Self-employed people can...",
  ...
]  # Each chunk ~500 tokens

Step 3: Embed Chunks
embeddings = [
  [0.23, -0.41, 0.15, ...],  # for chunk 1
  [0.19, -0.38, 0.22, ...],  # for chunk 2
  ...
]

Step 4: Store in Vector DB
vector_db.add(chunks, embeddings)
# Now ready for search!

Phase 2: Querying (Online, happens per request)

Python 
Step 1: Embed Query
user_query = "How do I maximize tax deductions?"
query_embedding = embed(user_query)

Step 2: Search Vector DB
similar_chunks = vector_db.search(query_embedding, k=5)
# Returns top-5 most similar chunks with scores

Step 3: Build Prompt
prompt = f"""
Answer the question using these documents:

{similar_chunks}

Question: {user_query}
Answer:
"""

Step 4: Call LLM
answer = llm.generate(prompt)
# LLM now has ground truth to answer from

Key insight: Indexing happens once (or periodically). Querying happens for every user request. Both are critical, but retrieval speed matters more for users.

Why RAG Works

RAG solves LLM hallucinations by providing grounding — anchoring answers in real documents. The LLM works better with context.

Without RAG (Hallucination Risk)

User: "What's the tax deduction limit for 2024?" LLM: "The limit is $15,000... (confidently wrong)" Problem: Model was trained on 2023 data, guesses about 2024.

With RAG (Grounded Answer)

User: "What's the tax deduction limit for 2024?" Retrieved: "2024 Tax Deduction Limits: $14,600 for single..." LLM: "The limit for 2024 is $14,600. (Based on retrieved docs)" Result: Accurate, up-to-date, cited sources.

RAG Tradeoffs

✓ Advantages

  • • Current information: Works with today's data, no retraining
  • • Sourceable: Cite which documents the answer came from
  • • Scalable: Add 1M documents without model changes
  • • Cost-effective: No expensive fine-tuning
  • • Reduces hallucinations: Answers grounded in facts

❌ Limitations

  • • Imperfect retrieval: Wrong documents retrieved first
  • • Latency: Search + LLM generation takes ~500-2000ms
  • • Context limits: LLM can only see top-K documents
  • • Cost per query: Embedding + LLM calls add up
  • • Still hallucinates: LLM invents facts despite good docs
